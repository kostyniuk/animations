Lesson 02 · How it's built

The animation loop

You know the push needs JavaScript because it's an integration of state. This lesson is the machine that does the integrating — the four-line heartbeat that every effect like grid.html runs on. Learn this one pattern and you can build all of them.

Mission: understand the techniques. This is the technique — everything else (forces, sprites, wrapping) is decoration hung on this loop.

01 The problem

You have state — a cube's position and velocity. Last lesson's rule says you must advance it yourself: position += velocity · dt. But that line only moves the cube once. To get motion you have to run it again, and again, forever, and repaint after each run. Two questions fall out:

  • When do I run it? A setInterval(fn, 16) seems obvious — but it fires whether or not the screen is ready to paint, fights the monitor's refresh, drains battery in background tabs, and stutters. You want to run exactly when the browser is about to draw a frame.
  • How much do I move per run? If you hardcode “move 3px each run”, your animation runs at one speed on a 60Hz laptop and nearly double on a 120Hz phone. The cube's speed becomes an accident of the user's hardware.

The animation loop answers both: requestAnimationFrame for when, and a measured delta time (dt) for how much.

02 The idea — a four-beat heartbeat

Ask the browser to call you right before it paints. Measure how long since last time. Advance the world by that much. Draw. Repeat.

let last = performance.now();
function loop(now) {
  requestAnimationFrame(loop);        // 1. book the next frame, always
  let dt = (now - last) / 1000;        // 2. seconds since last frame (~0.016)
  last = now;
  if (dt > 0.05) dt = 0.05;             //    clamp: tab was hidden, don't teleport
  update(dt);                          // 3. advance STATE: pos += vel*dt
  draw();                              // 4. repaint from current state
}
requestAnimationFrame(loop);             // kick it off

Four design decisions are hiding in there, and each one is a thing practitioners learned the hard way:

  1. rAF, not setInterval. The browser calls loop in sync with the display's refresh and pauses it entirely when the tab is hidden. You schedule the next frame at the top so the loop is self-sustaining. (MDN)
  2. Everything is multiplied by dt. Speed becomes “pixels per second”, not “pixels per frame”. Now 60Hz and 144Hz produce identical real-world speed — the frame just subdivides time more finely. (Gaffer On Games)
  3. Clamp dt. Switch tabs for 10 seconds and the next dt would be 10 — every object teleports across the screen. Capping dt trades a little accuracy for never exploding.
  4. update / draw are split. One function changes the world; the other only reads it to paint. Keep them separate and you can pause (stop calling update, keep calling draw), step one frame, or even run physics at a different rate than rendering.

Why dt isn't optional

Two fields, same cube, same speed. Left moves with x += speed · dt. Right moves a fixed x += step every frame. They're identical at full framerate. Now hit Throttle to simulate frames being dropped (a busy CPU, a heavy page):

x += speed · dt — correct
x += step — naive
mode 60fps
Throttled, the dt cube keeps its real-world speed — fewer frames, but each covers more time, so it stays in step. The fixed-step cube simply runs at half speed, because “per frame” silently became “per slower frame”. This is the bug that makes animations feel different on every machine — and why every line in grid.html's update ends in * dt.

The whole skeleton, annotated

This is the entirety of what drives the live demo above — setup, loop, and the two halves. Read it as the template for any canvas UI:

// --- SETUP (once) ---
const ctx = canvas.getContext('2d');
function resize() {                 // crispness: back the canvas with real device pixels
  const dpr = Math.min(devicePixelRatio, 2);
  canvas.width  = cssW * dpr;
  canvas.height = cssH * dpr;
  ctx.setTransform(dpr, 0, 0, dpr, 0, 0);  // now you draw in CSS pixels, sharp on retina
}

// --- UPDATE (advance state) ---
function update(dt) {
  for (const c of cubes) {
    if (pointer.active) {                 // distance-based force — the part CSS can't do
      const dx = c.x - pointer.x, dy = c.y - pointer.y;
      const dist = Math.hypot(dx, dy);
      if (dist < R) { const f = (1 - dist/R) ** 2 * PUSH;
        c.vx += dx/dist * f * dt; c.vy += dy/dist * f * dt; }
    }
    c.vx *= Math.exp(-3.2 * dt);          // damping = the spring-back "memory"
    c.vy *= Math.exp(-3.2 * dt);
    c.x += (DRIFT_X + c.vx) * dt;          // integrate: drift + push, scaled by dt
    c.y += (DRIFT_Y + c.vy) * dt;
    wrap(c);                               // re-enter from the opposite edge
  }
}

// --- DRAW (read state, paint) ---
function draw() {
  ctx.fillStyle = '#0a0a0b'; ctx.fillRect(0, 0, cssW, cssH);  // clear
  for (const c of cubes) drawCube(c.x, c.y);          // no layout cost — just pixels
}

That's it. Map it back to grid.html: its loop() is beat-for-beat this, uGrid() is this update, and its draw() is this draw with a prettier sprite. The seven modes are just seven different update functions sharing one loop.

04 Boundaries

  • Don't loop for a function of time. If state never accumulates — a constant spin, a one-shot fade — you don't need this machine; CSS (Lesson 01) is lighter and runs off the main thread.
  • The fixed-timestep upgrade. The variable dt here is perfect for visual effects. The moment physics must be deterministic or stable (games, springs that mustn't blow up), you separate a fixed-step update from a variable-rate draw and interpolate between them — the well-known “Fix Your Timestep” pattern. Same skeleton, smarter middle.
  • Next door — the force field itself. You've now seen the update body but not why those force/damping numbers behave like a spring. Lesson 03 opens up that vx += force·dt; vx *= exp(-k·dt) pair — the actual feel of the push.

05 Why this matters for your mission

You wanted to understand how these UIs are actually built. The honest answer is anticlimactic in the best way: there is no magic, there's one loop. requestAnimationFrame for timing, dt for fairness across hardware, and a clean split between changing the world and drawing it. Every cursor-grid, particle field, fluid sim, and physics toy you'll meet is this same four-beat heartbeat with a different update. Open grid.html, find loop = () => near the middle, and you'll now read it as old news — which is exactly the goal.
