Lesson 01 · The ambient layer

The infinite drift

The whole field of cubes in grid.html slides slowly, forever, with nobody touching it. That part needs no JavaScript at all — it's one of the most useful CSS tricks there is. This lesson is how it actually works, and you'll get to break it on purpose to see why it's built the way it is.

Mission: understand the techniques. We start with the half you can do in pure CSS, then mark exactly where CSS runs out and JS takes over.

01 The problem

You want a grid of cubes drifting endlessly across the screen. The obvious idea — put a row of cubes in a <div> and animate it moving — falls apart fast:

  • It runs out. A finite strip of cubes slides off and leaves empty space. To never run out you'd need infinite cubes.
  • The loop jumps. If you snap it back to the start to repeat, the viewer sees a jolt every cycle — the cubes teleport.

Before CSS animations, people faked this with looping GIFs or JS timers nudging scrollLeft. Both are clumsy. The clean answer is a single insight about repetition.

02 The idea — repeat, then slide by exactly one tile

Tile a pattern so it repeats forever, then animate it sliding by a whole number of tiles (one, two, however many). At the end of the slide the picture is pixel-identical to the start — so when the animation loops, there's nothing to see. Infinite motion from a 2-line keyframe.

Two CSS features do all the work:

  • A repeating background. background-image + background-repeat: repeat tiles one cube across the entire box, infinitely, for free. There are no “cubes” as elements — it's one painted texture.
  • Animating background-position. Sliding the texture's origin makes the whole tiled field move. (MDN: background-position)

The magic is the seamless-loop rule. The texture repeats every background-size pixels — here, every 70px. So position 0 and position 70px (or 140, 210…) look exactly the same. If your keyframe ends on a whole multiple of the tile in both axes, the last frame equals the first, and the jump back is invisible. Land between tiles and you get a visible hitch every loop. (MDN: Using CSS animations)

CSS only loop: seamless ✓

The entire source

This is all of it. Read the keyframe end values against the 70px tile:

.field {
  background-image: url(cube.svg);     /* SVG has transparent padding → built-in gap */
  background-size: 70px 70px;        /* one tile = 70×70 (cube fills ~58%, rest is gap) */
  background-repeat: repeat;       /* infinite cubes, for free */
  animation: drift 13s linear infinite;  /* linear! a constant glide, no ease */
}
@keyframes drift {
  from { background-position: 0 0; }
  to   { background-position: 140px 70px; } /* 2 tiles across, 1 down: whole multiples → seamless */
}
The thing that trips everyone up

The grid only ever moves 140px right and 70px down — then it snaps back to 0. That's the entire motion. The endless cross-screen drift is an illusion.

There are no cubes as elements. background-repeat paints one infinite wallpaper of cubes, and background-position slides that whole sheet as a single unit — from offset 0 to 140px 70px, then instantly back to 0, forever.

Why is the snap-back invisible? Because the wallpaper looks pixel-identical every 70px: offset 0 ≡ offset 70 ≡ offset 140. When it jumps 140 → 0, not a single pixel changes on screen, so your eye never catches the reset. Follow one cube and it travels 140px — but right as the sheet resets, an identical cube has slid into the spot it started from and carries the journey on. It's a treadmill: short belt, identical slats, looks endless. (A barber pole is the exact same trick.)

Three details that matter, and the reasons behind them:

  • Whole-tile endpoints (140 = 2×70, 70 = 1×70) → the seam is invisible. This is the rule you just broke.
  • The X:Y ratio is the drift angle. 140:70 = 2:1 = a gentle downward slope. Want steeper? 70:140. Want pure diagonal? 70:70. You can only pick angles that are ratios of whole tiles — the price of seamlessness.
  • linear timing → constant speed. The default ease would speed up and slow down within each loop, making the drift visibly “breathe”.

04 Boundaries — where CSS stops and JS begins

The drift works in CSS because it's a pure function of time: at any moment the offset is just speed × time, wrapped. CSS @keyframes is exactly that — hand it a timestamp, it returns a value, with no memory between frames. Perfect fit.

The deciding rule (carry this everywhere)

Is position a function of time (pos = f(t)) or an integration of state (pos += velocity·dt, with velocity itself changing each frame)? CSS can only ever do the first.

The cursor push in grid.html is the second kind: each frame a cube's velocity is built from last frame's velocity plus a force, then damped — that's why it has momentum and springs back. CSS has no per-frame hook to run that recurrence and no way to read “distance from this element to the pointer.”

You can still feel CSS's ceiling. Hover the cubes below: they lift and ease back (transitions can ease a state change!). But only the cube directly under the pointer reacts — no radial falloff to neighbours, no direction-aware shove, no momentum. CSS can answer “is the pointer on me?” but never “how far, and which way should I fling?”

CSS only The ceiling. Per-element :hover + transition — as far as CSS stretches toward interactivity, and it stops well short of the push.

05 Why this matters for your mission

You wanted to understand how the moving grid works — and now you can build that half yourself, from scratch, in two CSS rules, and explain why each number is what it is. Just as importantly, you've located the exact seam in grid.html: the ambient drift (the dx/dy constants) is a function of time that CSS could own, but it's fused on the same canvas as the pointer push (an integration of state) that CSS cannot. That fusion is why the real file is canvas, not CSS — and it's the perfect on-ramp to the loop that drives it.
